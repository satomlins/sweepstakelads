## Scoring (unchanged from 2024)

| Match outcome | Winner | Loser | GD counted? |
|---|---|---|---|
| Regular time | 3 | 0 | yes |
| After extra time | 3 | 1 | yes |
| Penalty shootout | 2 | 1 | **no** (match is technically a draw) |
| Draw (group stage only) | 1 | 1 | yes |

GS (goals scored) always counted from full-time score, ignoring penalty shootout goals.
Per-person totals = sum of their 4 teams' totals.

## Data source: Wikipedia (replacing football-data.org)

Same approach as `stremio_rugby/fixtures_index.py`:
- Fetch raw wikitext via `https://en.wikipedia.org/w/api.php?action=parse&prop=wikitext&page=<title>`
- Parse `{{footballbox}}` (and `{{footballbox collapsible}}`) template blocks
- Extract: date, time, home team, away team, scores, penalty scores, stage/round, status

**Pages to scrape:**
- `2026 FIFA World Cup group stage` тАФ all 12 group fixtures + standings
- `2026 FIFA World Cup knockout stage` тАФ R32 onwards
- (Fallback: per-group pages if the parent page is too large to parse cleanly)

**Why Wikipedia over the API:**
- No API key, no rate-limit anxiety
- Authoritative source updated within minutes of full-time
- Same pattern already proven in `stremio_rugby`
- Free for self-hosting on Oracle

**Risk:** wikitext template format can change. Mitigation: defensive parser, log unmatched matches, and have a `tests/` suite with snapshotted wikitext from a known good day.

**Separation of concerns** (a deliberate change from 2024 where it was all in one class):
- `scraper.py` тАФ only knows about Wikipedia/wikitext, returns plain dicts
- `scoring.py` тАФ pure functions, no I/O, takes match dicts and draw, returns standings
- `tournament.py` тАФ wires the two together, handles caching
- `app.py` тАФ Dash layout + callbacks only, no business logic

This makes the scoring rules unit-testable without hitting Wikipedia.

## Tables on the dashboard

Six tables, all with **team тЖТ person colour-coding wherever a team appears** (this is the key UX upgrade тАФ in 2024 only the Who column was coloured; now any team name carries its owner's colour):

1. **Person leaderboard** тАФ Who, PL, W, D, L, GD, GS, PNT (12 rows)
2. **Team table** тАФ Team, Owner, PL, W, D, L, GD, GS, PNT, In/Out (48 rows, sortable)
3. **Group tables** тАФ 12 mini-tables (one per group), 4 rows each, showing live group standings
4. **Upcoming fixtures** тАФ next ~10 matches, both teams coloured by owner
5. **Recent results** тАФ last ~10 finished matches, both teams coloured, scores, AET/pens annotation
6. **Knockout bracket** тАФ once group stage ends, a visual bracket with team owners colour-coded (stretch goal тАФ table form is acceptable for v1)

## Draw / pre-draw handling

- `draw_2026.csv` ships with **0 rows** initially (just headers `Who,Team`)
- App must render correctly with an empty draw: all team-owner cells display **"TBC"** (no colour), person leaderboard shows the 8+4 names with zeros
- Person table populated from a separate `participants.csv` (12 names тАФ 10 confirmed, 2 also "TBC") so the leaderboard exists before the draw
- Once drawn, `draw_2026.csv` gets 48 rows; "TBC" is replaced by owner names + colours; nothing else changes

## Caching & refresh

Same pattern as 2024 but tunable:
- Dash `Interval` triggers callback every 5 min
- If `last_updated.txt` < N minutes old, serve from CSV cache
- Otherwise re-scrape and rewrite CSVs
- N = 1 min during live matches, 30 min otherwise (stretch тАФ start with flat 5 min)

## Implementation phases

1. **Scraper** тАФ `scraper.py` + tests against snapshotted wikitext. No Dash, no scoring. Just: wikitext тЖТ list of match dicts.
2. **Scoring** тАФ `scoring.py` pure functions, fully unit-tested with the 2024 data as a regression test (we know what the final 2024 standings should look like).
3. **Tournament wiring** тАФ `tournament.py` ties scraper + scoring + caching. CLI-runnable: `python -m tournament` prints the current standings.
4. **Dash UI** тАФ new `app.py`, six tables, team-name colour-coding everywhere.
5. **Pre-draw state** тАФ verify the empty-draw case renders.
6. **Deploy** тАФ Oracle setup.

Each phase is mergeable independently. Phases 1тАУ3 can ship and run from the command line before any UI work.
