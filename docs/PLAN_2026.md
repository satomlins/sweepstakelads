# SWEEPSTAKELADS 2026 — Plan

Overhaul of the sweepstake dashboard for the 2026 FIFA World Cup. Branch: `wc2026`.

## Tournament context

- **Tournament:** 2026 FIFA World Cup (USA / Canada / Mexico, June 11 – July 19 2026)
- **Format:** 48 teams, 12 groups of 4 (A–L), top 2 + 8 best third-placed → Round of 32 → R16 → QF → SF → 3rd-place → Final
- **Sweepstake:** 12 participants × 4 teams = 48 (one team per participant per group is feasible but not enforced — depends how we run the draw)
- **10 participants confirmed:** Scott, Hugo, Sam, Brendan, Isaac, Adrian, Alex, Mary, Keshy, Jacob. **2 TBC.**
- **Draw status:** not yet held — schema must support a "no owner yet" state

## Scoring

| Match outcome | Winner | Loser | GD counted? |
|---|---|---|---|
| Regular time | 3 | 0 | yes |
| After extra time | 3 | 1 | yes |
| Penalty shootout | 2 | 1 | **no** (match is technically a draw, scores always equal) |
| Draw (group stage only) | 1 | 1 | yes |
| **Third-place match** (any means of victory) | **1** | **0** | **yes** |

GS (goals scored) and GA (goals against) always counted from full-time score (including AET), ignoring penalty shootout goals. GD is derived: GD = GS − GA.
Per-person totals = sum of their 4 teams' totals.

## Data source: Wikipedia (replacing football-data.org)

Same approach as `stremio_rugby/fixtures_index.py`:
- Fetch raw wikitext via `https://en.wikipedia.org/w/api.php?action=parse&prop=wikitext&page=<title>`
- Parse `{{footballbox}}` (and `{{footballbox collapsible}}`) template blocks
- Extract: date, time, home team, away team, scores, penalty scores, stage/round, status

**Pages to scrape:**
- `2026 FIFA World Cup group stage` — all 12 group fixtures + standings
- `2026 FIFA World Cup knockout stage` — R32 onwards
- (Fallback: per-group pages if the parent page is too large to parse cleanly)

**Why Wikipedia over the API:**
- No API key, no rate-limit anxiety
- Authoritative source updated within minutes of full-time
- Same pattern already proven in `stremio_rugby`
- Free for self-hosting on Oracle

**Risk:** wikitext template format can change. Mitigation: defensive parser, log unmatched matches, and have a `tests/` suite with snapshotted wikitext from a known good day.

## Architecture

```
sweepstakelads/
├── app.py                  # Dash app (UI + callbacks only)
├── tournament.py           # Tournament class — orchestrates scraping + standings
├── scraper.py              # Wikipedia scraping + footballbox parser
├── scoring.py              # Pure scoring functions (testable in isolation)
├── assets/
│   ├── draw_2026.csv       # Who → Team (populated after the draw)
│   ├── teamtable.csv       # Cached output
│   ├── persontable.csv     # Cached output
│   ├── fixtures.csv        # Cached output
│   └── last_updated.txt    # Cache timestamp
├── tests/
│   ├── fixtures/           # Snapshotted wikitext for parser tests
│   └── test_scraper.py
└── archive/                # Old 2024 code (see Cleanup)
```

**Separation of concerns** (a deliberate change from 2024 where it was all in one class):
- `scraper.py` — only knows about Wikipedia/wikitext, returns plain dicts
- `scoring.py` — pure functions, no I/O, takes match dicts and draw, returns standings
- `tournament.py` — wires the two together, handles caching
- `app.py` — Dash layout + callbacks only, no business logic

This makes the scoring rules unit-testable without hitting Wikipedia.

## Tables on the dashboard

Six tables, all with **team → person colour-coding wherever a team appears** (this is the key UX upgrade — in 2024 only the Who column was coloured; now any team name carries its owner's colour):

1. **Person leaderboard** — Who, PL, W, D, L, GD, GS, PTS (12 rows)
2. **Team table** — Team, Owner, PL, W, D, L, GD, GS, PTS, In/Out (48 rows, sortable)
3. **Group tables** — 12 mini-tables (one per group), 4 rows each, showing live group standings
4. **Upcoming fixtures** — next ~10 matches, both teams coloured by owner
5. **Recent results** — last ~10 finished matches, both teams coloured, scores, AET/pens annotation
6. **Knockout bracket** — once group stage ends, a visual bracket with team owners colour-coded (stretch goal — table form is acceptable for v1)

## Draw / pre-draw handling

- `draw_2026.csv` ships with **0 rows** initially (just headers `Who,Team`)
- App must render correctly with an empty draw: all team-owner cells display **"TBC"** (no colour), person leaderboard shows the 8+4 names with zeros
- Person table populated from a separate `participants.csv` (12 names — 10 confirmed, 2 also "TBC") so the leaderboard exists before the draw
- Once drawn, `draw_2026.csv` gets 48 rows; "TBC" is replaced by owner names + colours; nothing else changes

## Caching & refresh

Same pattern as 2024 but tunable:
- Dash `Interval` triggers callback every 5 min
- If `last_updated.txt` < N minutes old, serve from CSV cache
- Otherwise re-scrape and rewrite CSVs
- N = 1 min during live matches, 30 min otherwise (stretch — start with flat 5 min)

## Implementation phases

1. **Scraper** — `scraper.py` + tests against snapshotted wikitext. No Dash, no scoring. Just: wikitext → list of match dicts.
2. **Scoring** — `scoring.py` pure functions, fully unit-tested with the 2024 data as a regression test (we know what the final 2024 standings should look like).
3. **Tournament wiring** — `tournament.py` ties scraper + scoring + caching. CLI-runnable: `python -m tournament` prints the current standings.
4. **Dash UI** — new `app.py`, six tables, team-name colour-coding everywhere.
5. **Pre-draw state** — verify the empty-draw case renders.
6. **Deploy** — Oracle setup.

Each phase is mergeable independently. Phases 1–3 can ship and run from the command line before any UI work.
